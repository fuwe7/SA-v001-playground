# Roadmap: Анализатор баскетбольного броска

Цель: довести проект до **рабочего, полезного и доступного инструмента** —
снял видео → получил разбор техники, статистику и советы.

Принцип: двигаемся маленькими этапами. Каждый этап = что-то, что уже можно потрогать
и что работает само по себе. Не пытаемся сделать всё сразу.

---

## Важная развилка (про что нужно решить заранее)

Часть хотелок зависит от **трекинга мяча**, который пока отмечен как «тяжело»:
- «попал / не попал»
- «выпуск мяча в оптимальной точке»

Эти две фичи без мяча и кольца не сделать, поэтому они уезжают в поздний этап (Этап 5).
**Всё остальное (поза, углы, фазы, статистика, советы, эталон) можно сделать БЕЗ мяча** —
с этого и начнём, тут максимум пользы за минимум сложности.

Вторая развилка — **реалтайм vs загрузка видео**. Реалтайм на телефоне — самое сложное.
Сначала делаем «снял → загрузил → через несколько секунд отчёт» (это и есть миниаппка),
а реалтайм добавим позже как отдельный режим.

### Решение по ракурсу: «работает с любого, но честно»
Полностью «любой ракурс с одинаковой точностью» — нечестная обещалка: 2D-углы локтя/колена
точны только когда сустав видно сбоку. Поэтому делаем по-умному:
- Считаем углы не по плоской картинке, а по **3D-координатам MediaPipe (world landmarks)** —
  это заметно устойчивее к ракурсу.
- Программа **сама определяет ракурс** (сбоку / спереди / наискосок) и показывает только те
  метрики, которым в этом ракурсе можно доверять; остальные помечает «ненадёжно при таком ракурсе».
- В подсказках советуем «для лучшей точности снимай сбоку», но инструмент не падает с любого видео.

## Этап 0. Навести порядок (фундамент)
*Без новых фич. Цель: чтобы проект чисто запускался и с ним было приятно работать.*
- Почистить репозиторий: убрать ~80 МБ весов YOLO из git, убрать мёртвый/отладочный код.
- Починить зависимости (`requirements`) и написать нормальный README с инструкцией запуска.
- Свести логику в один модуль (источник правды). `app.py` пока убираем из фокуса.
- Понятная структура папок. Базовая связка «видео → скелет → углы» работает стабильно на обычном ПК (без GPU).
- Закрывает: «построить скелет», «импорт видео», «привести в порядок вообще всё».

## Этап 1. Полная биомеханика из позы (без мяча)
- Трекинг **обеих рук**, **туловища**, **бёдер**, **ног** (полный скелет + углы по всем ключевым суставам).
- Углы считаем по **3D world landmarks** + **авто-определение ракурса** (см. раздел про ракурс выше).
- **Время в воздухе** (по прыжку — вертикальное смещение бёдер/стоп).
- **Скорость выпуска** (скорость кисти в момент релиза).
- Уточнить фазы броска (присед → вынос → выпуск → приземление).
- Закрывает: вторая рука, туловище, ноги, бёдра, время в воздухе, скорость выпуска, «работает с любого ракурса».

## Этап 1.5. Простая веб-загрузка «видео → отчёт» (ранняя версия миниаппки)
- Минимальный экран: загрузил видео → получил скелет + метрики + графики.
- Цель — чтобы инструментом **можно было уже реально пользоваться** и давать обратную связь.
- Дальше этот же интерфейс обогащаем на Этапах 2–3 и допиливаем под телефон на Этапе 4.

## Этап 2. Статистика и графики
- Сохранение каждого броска со всеми метриками.
- Графики и сводная статистика по сессии.
- **Сравнение бросков** между собой.
- **«Отпечаток броска»** — профиль метрик одного броска как подпись/паттерн.
- Закрывает: статистика, графики, сравнение бросков, отпечаток.

## Этап 3. Качество броска, эталон и советы
- **Логика «хорошего броска»** (пороги/правила по метрикам).
- **Эталонный бросок** — записать образцовый и сравнивать остальные с ним.
- **Советы по коррекции** на основе отклонений («локоть недоразогнут», «нет работы ног» и т.п.).
- Закрывает: логика хорошего броска, эталон, советы по коррекции.

## Этап 4. Доступность — миниаппка с телефона
- Снял видео на телефон → загрузил → получил отчёт (биомеханика + статистика + советы).
- Простой и доступный интерфейс.
- Закрывает: миниаппка «снял и отправил», доступность.

## Этап 5. Мяч и попадания (тяжёлое, в конце)
- Трекинг мяча + кольца (YOLO).
- Дуга полёта, точка выпуска, **оптимальная точка релиза**.
- **Попал / не попал**.
- Закрывает: попал/не попал, оптимальная точка выпуска, трекинг мяча.

## Этап 6 (опционально). Реалтайм
- Анализ в реальном времени (на ПК с камерой, позже — на телефоне).

## Как этим управлять (для не-разработчика)
- Roadmap живёт прямо в репозитории файлом `ROADMAP.md` — всегда под рукой.
- Каждый этап разбиваем на маленькие задачи в **GitHub Issues**, группируем по **Milestones** (= этапы).
- Идём сверху вниз: один этап → один Pull Request (или несколько мелких).
- Код пишет Devin, ты ставишь цель и проверяешь результат.
